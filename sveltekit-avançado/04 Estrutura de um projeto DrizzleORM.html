Estrutura de um projeto DrizzleORM

Quando criamos um projeto SvelteKit com Drizzle, conforme a seção Criando um projeto do zero, ele vem estruturado da seguinte maneira:

📦 pasta-raiz
┣━━ 📂 drizzle
┣━━ 📂 src
┃   ┗━━ 📂 lib
┃       ┗━━ 📂 server
┃           ┗━━ 📂 db
┃               ┣━━ 📄 index.js
┃               ┗━━ 📄 schema.js
┣━━ 📄 .env
┣━━ 📄 drizzle.config.js
┗━━ 📄 package.json

Cinco importantes arquivos são criados:

O arquivo package.json

Além dos pacotes mencionados no tutorial Criando um projeto SvelteKit, um projeto com Drizzle inclui:

  • dependencies (Dependências usadas no ambiente de produção):
    • drizzle-orm: Biblioteca de mapeamento objeto-relacional (ORM) usado para interagir com bancos de dados SQL. Ele fornece uma API baseada em SQL, permitindo escrever consultas seguras e eficientes.
    • @libsql/client: Um driver para JavaScript que permite conectar-se a bancos de dados SQLite.
  • devDependencies (Dependências usadas apenas no ambiente de desenvolvimento):
    • drizzle-kit: Ferramenta CLI para gerenciamento de migrações no DrizzleORM. Permite gerar e aplicar migrações no banco de dados com comandos simples, além de introspecção de esquemas.

Além dos comandos SvelteKit definidos no arquivo package.json, também estão inclusos os seguintes comandos do DrizzleKit:

  • pnpm run db:push:

    • Aplica o esquema do Drizzle diretamente no banco sem gerar um arquivo de migração.
    • Uso comum: Atualizar o banco rapidamente sem versionamento de mudanças.
    • Cuidado: Pode sobrescrever alterações existentes no banco irreversivelmente!

  • pnpm run db:migrate:

    • Aplica migrações pendentes ao banco de dados.
    • Antes de rodar esse comando, é necessário gerar as migrações (pnpm exec drizzle-kit generate).
    • Uso comum: Manter um histórico seguro das mudanças na estrutura do banco.

  • pnpm run db:studio:

    • Inicia um servidor web local com interface visual interativa para explorar e gerenciar o banco de dados.
    • Permite visualizar tabelas, executar consultas e depurar dados diretamente pelo navegador.
    • Uso comum: Inspecionar ou modificar dados manualmente sem usar programação ou SQL diretamente.

Se precisar modificar o banco de dados, use pnpm run db:migrate para versionar mudanças ou pnpm run db:push para sincronizar diretamente. Caso queira inspecionar os dados, o pnpm run db:studio pode ser útil.

O arquivo .env

O arquivo .env (dotenv) é um arquivo de texto usado para armazenar variáveis de ambiente em projetos de software. Ele permite manter informações sensíveis e configurações de ambiente separadas do código-fonte, facilitando a configuração do projeto e aumentando a segurança. Suas principais características incluem:

  • Armazena credenciais e informações sensíveis
    • Exemplos: chaves de API, tokens de autenticação, configurações de banco de dados.
  • Facilita a configuração de diferentes ambientes
    • Permite definir valores específicos para ambientes de Desenvolvimento (.env.dev), Teste (.env.test) e Produção (.env.prod).
  • Evita "código chumbado" (hard-coded)
    • Alterações de configuração podem ser feitas sem modificar o código-fonte, reduzindo o risco de erros e a necessidade de recompilação.
Observação: O termo hard-coded (código chumbado) se refere a valores ou configurações inseridos diretamente no código-fonte de um programa, em vez de serem armazenados em arquivos externos, variáveis de ambiente ou banco de dados. Isso torna o código menos flexível e mais difícil de manter.

Como o arquivo .env pode conter informações sensíveis e configurações de ambiente específicas, ele não deve ser versionado. Para evitar vazamentos, deve ser incluído no arquivo .gitignore.

No caso de projetos com Drizzle e SQLite, crie na pasta raiz o arquivo .env:

DATABASE_URL=file:meubanco.db

A variável de ambiente DATABASE_URL define o caminho do banco de dados SQLite que será utilizado pelo Drizzle ORM. O valor file:meubanco.db indica que o SQLite armazenará os dados no arquivo meubanco.db, localizado na pasta raiz.

Diferente de bancos como PostgreSQL ou MySQL, que utilizam um servidor, o SQLite é um banco de dados local e baseado em arquivos. O Drizzle abre o arquivo meubanco.db onde são armazenadas as tabelas e os dados. Se meubanco.db não existir, o SQLite cria automaticamente esse arquivo com um novo banco de dados.

O arquivo schema.js

No Drizzle ORM, o arquivo schema.js define a estrutura do banco de dados. Em projetos SvelteKit, ele geralmente fica localizado em src/lib/server/db/. Esse arquivo descreve as tabelas, colunas, tipos de dados e restrições aplicadas ao SQLite.

Inicialmente, ele pode conter algumas tabelas de exemplo, mas depois excluiremos essas tabelas e definiremos nosso próprio esquema de dados.

O arquivo index.js

O arquivo index.js, localizado em src/lib/server/db/, é responsável por configurar a conexão com o banco de dados no Drizzle ORM. Ele inicializa o cliente de banco de dados (quem se conecta ao banco) e exporta um objeto que representa o banco (db), que será usado em todo o aplicativo para acessar o banco.

Explicação do código

import { drizzle } from 'drizzle-orm/libsql';
import { createClient } from '@libsql/client';
import * as schema from './schema';
import { env } from '$env/dynamic/private';
  • drizzle-orm/libsql: Importa a função drizzle, usada para criar a conexão com o banco de dados.
  • @libsql/client: Importa createClient, que cria o cliente de conexão com o banco, através do driver LibSQL.
  • ./schema: Importa o esquema do banco de dados definido no arquivo schema.js.
  • $env/dynamic/private: Importa variáveis de ambiente privadas do SvelteKit. 
if (!env.DATABASE_URL) throw new Error('DATABASE_URL is not set');
  • Verifica se a variável de ambiente DATABASE_URL está definida. Se não estiver, lança um erro para evitar que a aplicação tente rodar sem uma conexão configurada. 
const client = createClient({ url: env.DATABASE_URL });
  • Cria um cliente do banco de dados usando a URL definida na variável de ambiente DATABASE_URL. 
export const db = drizzle(client, { schema });
  • Cria e exporta o objeto de ORM, associando o cliente do banco e o esquema (schema).
  • Esse objeto db será utilizada no código para interagir com o banco de dados, permitindo consultas, inserções, atualizações e exclusões sem precisar escrever SQL diretamente.

O arquivo drizzle.config.js

O arquivo drizzle.config.js configura o funcionamento do Drizzle Kit. O Drizzle Kit é um kit de programas CLI (Command Line Interface) que gerencia migrações do banco de dados.

Basicamente, para gerenciar migrações em um banco de dados SQLite com o Drizzle Kit, as principais propriedades de configuração são:

  • schema: Caminho do arquivo que define as tabelas e colunas do banco (schema.js).
  • dbCredentials: Define as propriedades de conexão do banco de dados (no caso de SQLite, precisamos apenas da URL, contida na variável de ambiente DATABASE_URL).

Além dessas propriedades obrigatórias para se conectar a um banco de dados SQLite, o arquivo de configuração pode ter várias outras propriedades, dentre elas:

Propriedade Descrição
verbose Se true, exibe mais detalhes ao rodar comandos no terminal.
strict Se true, ativa verificações mais rigorosas no esquema do banco.
dialect Define o tipo de banco de dados usado (sqlite, no caso).

A Pasta drizzle

A pasta drizzle é onde ficam armazenados os arquivos de migração gerados pelo Drizzle Kit. Ela mantém um histórico das alterações feitas no esquema do banco de dados, garantindo que as mudanças possam ser aplicadas, revertidas ou rastreadas. Em resumo, cada migração representa uma "versão" do banco, permitindo evoluí-lo sem perder dados.

Depois de rodar um comando de geração de migração como...

pnpm exec drizzle-kit generate

... A pasta drizzle/ será criada (caso ainda não exista) e conterá arquivos como:

  • 0000_initial_migration.sql: Primeira migração, criando tabelas do schema.
  • 0001_add_column_users.sql: Exemplo de uma migração que adiciona uma coluna
  • 0002_create_orders_table.sql: Exemplo de uma migração para criar uma tabela

Esses arquivos contêm comandos SQL que o Drizzle usará para modificar o banco.

Para aplicar as migrações geradas, basta rodar o comando: 

pnpm run db:migrate

Isso executará os arquivos .sql na sequência correta, garantindo que todas as mudanças sejam refletidas no banco de dados.
